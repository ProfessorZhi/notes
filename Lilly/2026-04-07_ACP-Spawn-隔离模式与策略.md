# ACP Spawn 隔离模式与策略

## 核心规则

OpenClaw 的 ACP Spawn 有两个关键隔离维度：

### 1. 运行时隔离（Runtime Isolation）

| 请求者 | 可 Spawn | 原因 |
|--------|---------|------|
| Host 会话 | `runtime="acp"` | ACP 运行在 Host |
| Sandboxed 会话 | `runtime="subagent"` | Subagent 运行在沙箱内 |

**关键约束**：沙箱会话不能 Spawn ACP 会话，因为 ACP 运行在 Host 层面，超出沙箱边界。

### 2. 会话模式（Spawn Modes）

```typescript
ACP_SPAWN_MODES = ["run", "session"]
```

- `run`：一次性任务，执行完即结束
- `session`：持久会话，线程绑定，支持后续跟进

### 3. 沙箱模式（Sandbox Modes）

```typescript
ACP_SPAWN_SANDBOX_MODES = ["inherit", "require"]
```

- `inherit`：继承父会话的沙箱状态
- `require`：强制要求沙箱环境

## 策略检查链

Spawn 前会检查：
1. 请求者是否为沙箱会话 → 沙箱会话拒绝 ACP spawn
2. `sandbox="require"` 但当前为 ACP runtime → 矛盾拒绝
3. 心跳策略是否开启
4. 线程上下文是否可用

## 工程教训

**多 Agent 系统中，运行时边界决定了能访问哪些资源。**
沙箱隔离不仅是安全边界，也是能力边界——沙箱内 Agent 无法直接启动 Host 级别的服务。

这个模式对设计多 Agent 分层有重要意义：
- 主管 Agent（Supervisor）通常在 Host 层运行
- 执行 Agent（Worker）可以在沙箱中运行
- 跨层调用需要通过明确的 IPC/消息机制，而不是直接 spawn
