# Claude Code Best Practices 核心总结

## 信息来源
- https://code.claude.com/docs/en/best-practices

## 核心理念

Claude Code 是 agentic 编码环境：不是审代码的聊天机器人，而是能读文件、跑命令、改代码、自主工作的 agent。

核心约束：上下文窗口填满速度快，性能随上下文填充下降。

## 最重要的一点：给 Claude 一个验证自己工作的方式

这是单次交互中杠杆最高的事。

| 策略 | 差的做法 | 好的做法 |
|------|---------|---------|
| 提供验证标准 | 写一个验证 email 的函数 | 提供具体测试用例，实现后运行测试 |
| 视觉验证 UI | 让 dashboard 更好看 | 截图 + 描述设计，实现后截图对比差异 |
| 解决根因 | build 失败了 | 贴错误信息，要求解决根因而非压制错误 |

没有验证标准时，Claude 可能产出看起来对但实际不工作的东西。

## 推荐工作流：探索 → 规划 → 实现 → 提交

1. **Plan Mode 探索**：Claude 只读文件，不修改
2. **Plan Mode 规划**：创建详细实现计划
3. **Normal Mode 实现**：按计划实现，边做边验证
4. **提交**：描述性 commit message + PR

Plan Mode 有成本——适合范围不明确或多文件修改的场景。
小改动（typo、log 行、变量重命名）直接做，不需要 plan。

## 上下文管理是关键约束

- 文件读取主导 token 消耗
- 用 subagent 处理研究密集型任务
- 上下文快满时性能下降

## 工程意义

1. **验证机制是 agent 可靠性的核心**：没有自验证的 agent 等于每次错误都要人工发现
2. **Plan Mode 是质量门控**：探索和执行分离，避免做错需求
3. **小任务直接做不过度设计**：能用一句话描述 diff 的不需要 plan
4. **上下文预算是根本约束**：管理 token 消耗比优化 prompt 更有效
