# OpenClaw 重试与队列机制笔记\n\n## 重试策略\n\n- 默认 3 次重试，max delay 30s，jitter 10%\n- 按 HTTP 请求级别重试，不是整个流程\n- 只重试 transient errors（429 timeout connect/reset）\n- 非幂等操作不重试（避免重复）\n- Markdown 解析失败会 fallback 到纯文本，不重试\n\n## 命令队列设计\n\n目的：序列化入站 auto-reply runs，防止多个并发 run 碰撞共享资源。\n\n机制：\n- lane-aware FIFO，按 session key 分 lane\n- 全局 lane 控制整体并发（main 默认 4，subagent 默认 8）\n- typing indicators 在入队时立即触发，体验不受影响\n\n队列模式：\n- collect（默认）：合并所有消息为一个 followup turn\n- steer：立即注入当前 run\n- followup：等当前 run 结束后排队\n- steer-backlog：steer 并保留 followup\n\n## 故障模式联想\n\n上次 obsidian_ops.py exec 失败（SIGKILL）：shell 把 JSON5 内容当命令执行。\n结论：向 obsidian_ops.py 传 content 时，避免使用会被 shell 扩展的字符（如大括号、冒号等），或确认 python 直接接收参数而不是经过 bash。\n\n## 工程价值\n\n- 调试时：看到 "queued" 日志说明等待超过 2s\n- 配置时：maxConcurrent 控制全局并发，session lane 并发由 session key 隔离\n- 重试配置：对 Telegram/Discord 可分别调优\n\n## 来源\n\n/home/dministrator/openclaw-wsl/docs/concepts/retry.md\n/home/dministrator/openclaw-wsl/docs/concepts/queue.md\n
