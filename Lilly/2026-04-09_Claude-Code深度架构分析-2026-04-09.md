# Claude Code 深度架构分析（wavespeed.ai）

## 信息来源

wavespeed.ai: "Claude Code architecture Deep Dive: What the Leaked Source Reveals"

## 技术栈

- Runtime: **Bun**（不是 Node.js）
- TUI: React + Ink（用 React 组件模式做 CLI）
- Schema 验证: **Zod v4**

## 工具系统：40+ 独立权限隔离模块

每个工具有独立的：
- 输入 schema
- 权限级别
- 执行逻辑

**没有跨工具共享可变状态。**

BashTool 和 FileReadTool 在同一注册表，但风险模型完全不同：
- BashTool → 可以修改系统状态
- FileReadTool → 只读

架构为每个工具独立 gating，而不是用统一策略。

**AgentTool = subagent 就是普通工具调用**
sub-agent 是注册表中的第一等公民，spawn sub-agent 就是一次普通 tool call，不需要单独的编排层。这让架构非常扁平。

## QueryEngine.ts — 46K 行的单一文件

所有 LLM API 调用、流式输出、缓存、编排都在**一个文件**里。

理由：与模型 API 相关的所有逻辑（重试、限流、上下文预算、流式错误）集中在一处，便于统一推理。

**自愈查询循环**：
当上下文预算接近上限时，引擎不会崩溃或请求帮助——自动触发压缩，在触及上限前预留 buffer，生成结构化摘要。这是设计行为，不是 hack。

## 多 Agent 编排：Mailbox Pattern

worker agent 执行危险操作时：
1. **不能**自行批准
2. 向 coordinator 的 mailbox 发请求
3. 等待 coordinator 评估并批准/拒绝

**原子 claim 机制**：防止两个 worker 同时处理同一审批——并行执行时的竞态控制。

**共享内存空间**：所有 agent 共享上下文，避免重复获取。

这是与 naive multi-agent 设计的本质区别——coordinator/worker 分层 + 审批门控实现并行但不混乱。

## 三层上下文压缩（最有直接工程价值）

| 层级 | 触发时机 | 机制 | API 成本 |
|------|---------|------|---------|
| **MicroCompact** | 持续进行 | 直接修剪旧工具输出，不调用 API | **零成本** |
| **AutoCompact** | 接近 context 上限时 | 预留 13K token buffer，生成最多 20K token 结构化摘要 | 有成本 |
| **Full Compact** | 彻底耗尽前 | 完整压缩对话，重新注入最近访问文件（每文件最多 5K token）、活跃 plan、相关 skill schema | 有成本 |

**内置断路器**：连续 3 次压缩失败后停止重试——防止无限循环。

**工作预算重置**：Full Compact 后工作预算恢复到 50K token。

## Feature Flags：108 个模块不存在于生产版本

Bun 在编译时通过 dead code elimination 消除标记关闭的代码。生产版本永远干净。

示例：KAIROS、VOICE_MODE、DAEMON 在源码中存在，但外部构建中被消除。

Undercover Mode（在 git commit 中隐藏内部代号）这个设计在泄露源码中自己泄露了——关于软件交付管道中风险积累的教科书案例。

## Telemetry：两个有意义的指标

1. **Frustration Metric**：追踪用户骂人的频率——UX 崩溃的先行指标
2. **Continue Counter**：追踪用户输入 "continue" 的频率——agent 停滞、用户需要干预的指标

这些不是虚荣指标，是标准分析会漏掉的特定失败模式。

## 工程意义

1. **subagent as tool call 是优雅但需要谨慎的设计**：每个有 AgentTool 权限的 agent 都能 spawn 无限子 agent，需要在工具级别做权限隔离
2. **三层压缩是长程会话的必备工程**：不管理上下文预算的 agent 工具在规模上会静默失败或硬报错
3. **Mailbox Pattern 是多 Agent 并行安全的关键**：危险操作必须走 coordinator 审批，而不是自治
4. **自愈查询循环减少人工干预**：QueryEngine 自动处理限流和压缩，用户不需要盯着
5. **frustration/continue telemetry 是值得借鉴的 UX 监控思路**：中国区 agent 产品可以用类似指标监控真实体验
