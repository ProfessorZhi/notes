# Claude Code 缓存与性能模式

## 1. Prompt Cache 稳定性问题

### Tool Schema Bytes 作为 Cache Key

Claude Code 使用 tool schema bytes 作为 prompt cache key 的一部分。**问题**：tool schema bytes 在 session 期间变化会导致 prompt cache miss。

导致 schema bytes 变化的场景：
- MCP server reconnect（MCP tool/resource cache leak on server reconnect，v2.1.84 修复）
- Dynamic tool 加载/卸载
- Tool description 更新

**工程意义**：如果要实现稳定的 prompt cache，必须锁定 tool schema。如果 schema 会动态变化，需要有版本策略或 cache key 的更细粒度设计。

### Prompt Cache 对话历史

Claude Code 2.1.92 为 Pro 用户添加了返回后的提示：当 prompt cache 过期后，显示下次 turn 将发送多少 uncached tokens。这说明 prompt cache 的管理是显式可感知的。

## 2. StructuredOutput Schema Cache Bug

Claude Code 2.1.89 修复：使用多 schema 时 ~50% 失败率。

Bug 原因未公开，但涉及：
- 多个 schema 同时使用时的缓存失效问题
- 可能是 LRU cache 与 schema 版本的交互问题

**工程意义**：多 schema 场景下的缓存一致性是容易出问题的点。如果设计多 tool/schema 系统，需要注意缓存的 invalidation 策略。

## 3. JSON Input LRU Cache Leak

Claude Code 2.1.89 修复：large JSON inputs 被 retained 作为 LRU cache keys，导致内存泄漏。

**问题**：JSON 输入作为 cache key 但没有被正确 evict。

**工程意义**：如果系统使用 LRU cache 缓存大型对象，需要确保：
- cache size 有明确上限
- eviction 策略正确实现
- 大型对象的引用计数正确管理

## 4. Per-turn JSON.stringify 开销优化

Claude Code 2.1.89 优化：消除了每 turn 对 MCP tool schemas 的 JSON.stringify 操作，提升性能。

**问题**：MCP tool schemas 每 turn 都被序列化，即使没有变化。

**工程意义**：Tool schema 的序列化应该是增量更新或缓存的，而不是每 turn 全量序列化。这对 tool 数量多的场景影响大。

## 5. MCP Schema Caching

Claude Code 使用 `enabledMcpjsonServers` 配置（如 `["MiniMax"]`）。

MCP tool descriptions 和 server instructions 被 cap 在 2KB（v2.1.84），以防止 OpenAPI-generated servers 膨胀 context。

## 6. 其他性能优化

| 版本 | 优化项 |
|------|--------|
| 2.1.88 | Write tool diff computation 60% faster（对 tabs/`&`/`$` 处理） |
| 2.1.86 | `@` mention 时 raw string content 不再 JSON-escaped，减少 token 开销 |
| 2.1.86 | Skill descriptions 在 `/skills` 列表中被 cap 在 250 字符 |
| 2.1.84 | Startup 改进 ~30ms（parallel setup()） |
| 2.1.84 | p90 prompt cache rate 改进 |
| 2.1.79 | Startup 内存减少 ~18MB |
| 2.1.80 | 大仓库（250k files）启动内存减少 ~80MB |

## 工程意义总结

1. **Prompt cache 稳定性 > 性能**：先保证 cache key 稳定，再优化序列化开销
2. **Tool schema 管理是核心**：MCP dynamic discovery 带来 cache invalidation 复杂度
3. **大对象 cache 需要有明确边界**：LRU cache size limit 是必须的
4. **Schema 版本策略**：如果 schema 会变化，需要有版本化和 cache key 分离的策略
