# 多 Agent 主管回路：耐久的工程模式

## 来源

`openclaw_wsl/20-Bot工作流/主管Bot复盘回路-MVP-2026-04-03.md`

## 核心经验

### 硬上限轮次（最重要）

每个任务内部最多处理 **3 轮**：

1. 首次执行
2. 第一次返工或改派
3. 第二次返工或改派

第 3 轮仍不达标 → 停止循环，不做无限死磕。

**为什么这耐得住：**
- 防止 token 成本失控
- 防止延迟累积
- 防止责任链模糊
- 给用户一个可预测的交付节奏

### 返工 vs. 改派（实用区分）

| 情况 | 动作 | 原因 |
|------|------|------|
| Bot 做错了 | 返工（仍派给原 bot）| 能力够，只是没做到位 |
| Bot 做对了但不适合 | 改派（换人）| 人找错了，不是做得差 |

典型链条：`绮雯` 查到信息 → 改派 `Sasha` 整理 → 改派 `Vivian` 执行飞书

**工程价值：** 把"人错了"和"做得差"分开处理，避免强迫 bot 做不适合的事。

### 轻量验收原则

主管不做长篇复盘，只查：

- 是否真的完成了动作
- 是否返回了可用结果
- 是否给出了链接、对象、状态等可交付物

不追求"世界最佳答案"，只追求"够交付"。

### 可见/不可见边界（测试阶段）

**公开：** 指派、结果、返工理由、改派理由、最终收口
**隐藏：** 原始长命令输出、大段工具日志、低价值报错堆栈

## 工程适用场景

- 多 bot 协作的工作流设计
- 防止无限循环的机制设计
- 主管/执行者角色分离的设计
- 任务分工（Vivian=执行，Sasha=整理，绮雯=调研）
