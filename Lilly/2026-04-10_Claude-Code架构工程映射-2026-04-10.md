# Claude Code 架构工程映射：设计决策对照表

## 时间

2026-04-10

## 核心发现

整理近期 Claude Code 源码研究，将设计决策按"背景 → 决策 → 工程价值"格式整理。

---

### 1. 工具并发策略

**背景**：Agent 常需同时调用多个只读工具（搜索、文件读），但写操作并发会导致状态冲突。

**决策**：partitionToolCalls 将工具分为只读/非只读，只读最多 10 个并发，非只读串行。

**工程价值**：
- 设计 multi-tool agent 时，第一步就是区分 readonly/write
- 只读工具并发可显著提升速度（10x）
- 写操作串行是正确性优先的体现

---

### 2. 任务状态模型

**背景**：Claude Code 的 task 不是"函数调用后返回"，而是"有生命周期对象"。

**决策**：Task 有 pending/running/completed/failed/killed 状态，支持 pause/kill，输出持久化到文件。

**工程价值**：
- 有状态任务支持中断恢复（断点调试、超时处理）
- outputFile 模式适合长任务输出流（避免内存溢出）
- totalPausedMs 精确记录有效执行时间
- 对比：OpenClaw agent loop 更像"事件驱动函数"，Claude Code 更像"有状态后台进程"

---

### 3. 权限模型：deny-list 而非 allow-list

**背景**：如果用 allow-list，每新增一个工具都要手动加到白名单，工作量巨大。

**决策**：disallowedTools 只禁用 4-5 个高危工具（AGENT_TOOL、FILE_EDIT 等），其余默认可用。

**工程价值**：
- deny-list 是最小权限原则的更实用版本
- 只需管住真正危险的操作（spawn子agent、写文件、执行命令）
- allow-list 适合极高安全要求但维护成本高
- China 区 agent 工程可借鉴：deny-list 降低配置成本

---

### 4. Verification Agent 的强制证据要求

**背景**：只读代码审查无法保证代码真正可用，LLM 容易"被前 80% 迷惑"。

**决策**：Verification agent 必须包含命令执行证据（**Command run:** + **Output observed:**），输出格式化为 VERDICT: PASS/FAIL/PARTIAL。

**工程价值**：
- "代码看起来对"不等于"代码真正可用"
- 强制命令执行证据是验收的最可靠形式
- 对 China 区 LLM 应用工程：代码验收必须要求实际运行证据
- 对抗性探测（并发、边界值、幂等性）是高质量验证的核心

---

### 5. Multi-agent 上下文策略

**背景**：子 agent 需要从父 agent 继承上下文，但完全复制效率低。

**决策**：
- Coordinator 模式：专门 orchestrator 拆解分发，共享 context
- Fork 模式：完整上下文继承，但 recursive forking 被禁止（检测 fork boilerplate tag）

**工程价值**：
- Fork 适合"高度协作、上下文强共享"场景
- OpenClaw 每个 agent 独立 session，适合"独立 agent、消息路由"场景
- recursive forking 禁止防止上下文无限膨胀
- China 区多 bot 方案（nami 主管）更接近 OpenClaw 路由模型

---

## 核心模式总结

| 设计维度 | Claude Code 做法 | 对应工程价值 |
|---------|-----------------|-------------|
| 工具并发 | readonly 10并发 / write 串行 | 多读少写场景提速 |
| 任务模型 | 有状态 + pause/kill/file输出 | 长任务可靠性 |
| 权限策略 | deny-list（4-5个高危工具） | 最小权限实用版 |
| 验收标准 | 强制命令执行证据 + VERDICT | 防止"看起来对" |
| 上下文 | Fork全量继承 / Coordinator分发 | 按场景选共享策略 |
