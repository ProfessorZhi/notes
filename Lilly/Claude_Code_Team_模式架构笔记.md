# Claude Code Team 模式架构笔记

## 核心概念

Claude Code Team 模式是一种多智能体协作方案，与旧的 SAG（Single Agent Group）模式本质不同：

| 特性 | SAG 模式 | Team 模式 |
|------|---------|-----------|
| 上下文持久性 | 每次启动重置 | 子智能体持续待命 |
| 上下文管理 | 200K 上下文容易溢出 | 任务拆分到各子智能体，分摊压力 |
| 性能衰减 | 压缩后急剧下降 | 上下文分配到各 Agent，整体稳定 |
| Token 消耗 | 基准水平 | 约为 SAG 的 2 倍及以上 |

## 推荐的 6 子智能体团队配置

| 角色 | 职责 |
|------|------|
| 后端开发 (Backend Dev) | 后端功能开发 |
| 前端开发 (Frontend Dev) | 前端界面开发 |
| 研究员 (Researcher) | 需求调研、技术调研 |
| 端到端测试 (E2E Tester) | 测试用例编写与执行 |
| 审查员 (Reviewer) | 代码审查 |
| 重构清洁器 (Refactorer) | 代码重构优化 |

## 文件持久化系统

每个子智能体维护自己的文档集合：
- `progress.md`：当前进展
- `plan.md`：执行计划
- `findings.md`：发现和调研结果

项目级文档结构：
```
项目目录/
├── leader/          # 主智能体文档
├── backend/         # 后端开发文档
├── frontend/        # 前端开发文档
├── researcher/      # 研究员文档
└── [任务文件夹]/    # 具体任务创建独立子文件夹
```

## 工程意义

1. **上下文分摊**：200K 上下文任务被分摊到多个子 Agent，避免单 Agent 溢出
2. **透明可见**：用户可以实时查看每个角色的进展（通过文件）
3. **独立追踪**：每个角色维护自己的文档集合，不依赖 Agent 内存状态
4. **成本换性能**：Token 消耗翻倍，但换来了上下文管理的稳定性

## 使用建议

- 小型项目（200K 上下文足够）：不需要 Team 模式
- 中大型项目（上下文经常溢出）：强烈推荐 Team 模式
- Claude Code Team 模式的核心价值：将超出上下文的复杂任务拆解到多个子智能体中
