# Claude Code 三大扩展机制对比

## 信息来源

- https://code.claude.com/docs/en/features-overview
- https://code.claude.com/docs/en/hooks-guide
- https://code.claude.com/docs/en/skills
- https://code.claude.com/docs/en/agent-teams

## 扩展机制全景图

```
Hook（确定性脚本）
  ↓ 
Skill（LLM 可调用的知识/工作流）
  ↓
Subagent（隔离上下文的工作单元）
  ↓
Agent Team（多独立 session 协作）
```

## 1. Hooks — 确定性脚本，LLM 不参与

Hook 是在 Claude Code 生命周期**特定事件**上运行的 shell 命令。
- 无 LLM 调用，结果确定
- 在 agent loop **外部**执行
- 可 block 行为（exit code 2 = block）

### 典型事件

| 事件 | 时机 | 典型用途 |
|------|------|---------|
| `PostToolUse` | 工具执行后 | 自动格式化、lint |
| `Edit|Write` (matcher) | 写/编辑文件后 | Prettier 格式化 |
| `PreToolUse` | 工具执行前 | 阻止危险操作 |
| `Notification` | Claude 等待输入时 | 桌面通知 |
| `TaskCreated` | task 创建时 | 精确拦截 |
| `Match` | session 启动时 | 重新注入 context |

### 关键设计：Conditional Hooks

支持 `if` 字段 + permission rule 语法（如 `Bash(git *)`）：
```json
{
  "matcher": "Bash",
  "if": "Bash(git *)",
  "hooks": [{ "type": "command", "command": "..." }]
}
```

这比全局 hook 更高效，只在匹配条件时触发。

### 工程意义

- **自动化合规**：lint/format 在每次编辑后自动运行，不需要 LLM 判断
- **安全门控**：PreToolUse block 危险命令（如删除 .git）
- **零 LLM 成本**：确定性子集不应该浪费模型调用

## 2. Skills — LLM 可调用的知识和工作流

Skill 是一个 `SKILL.md` 文件，包含：
- YAML frontmatter（name + description）
- markdown 内容（供 LLM 读取和执行）

### 两种加载方式

1. **用户调用**：`/skill-name` 直接触发
2. **自动加载**：Claude 根据 description 决定何时加载

### 关键设计：按需加载

Skill 内容**只在调用时**加载到上下文，不调用就不占 token。
这解决了"长参考文档撑爆上下文"的问题。

### 存储位置（优先级从高到低）

| 位置 | 路径 | 范围 |
|------|------|------|
| Enterprise | managed settings | 组织级 |
| Personal | `~/.claude/skills/<name>/SKILL.md` | 用户级 |
| Project | `.claude/skills/<name>/SKILL.md` | 项目级 |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | 插件启用处 |

### Skill 目录结构

```
my-skill/
├── SKILL.md           # 主指令（必需）
├── template.md        # Claude 填写的模板
├── examples/
│   └── sample.md      # 示例输出
└── scripts/
    └── validate.sh    # Claude 可执行的脚本
```

### 与 OpenClaw Skill 对比

| 维度 | Claude Code Skill | OpenClaw Skill |
|------|------------------|---------------|
| 格式 | Markdown + YAML frontmatter | SKILL.md |
| 调用方式 | `/name` 或自动加载 | 触发条件 |
| 上下文占用 | 按需加载（不调用不占） | 启动时加载快照 |
| 支持文件 | 目录结构 + 脚本 | 无明确结构 |

## 3. Subagent — 隔离上下文的工作单元

Subagent 是独立上下文窗口的工作单元：
- 有自己的完整上下文
- 不能嵌套 spawn subagent
- 返回摘要，不撑爆主对话

### 与 Skill 的区别

| | Skill | Subagent |
|--|-------|---------|
| LLM 调用 | 在主上下文中执行 | 独立上下文窗口 |
| 上下文隔离 | 无（共享主上下文） | 有（完全隔离） |
| 用途 | 知识/工作流 | 长时间并行任务 |
| token 成本 | 低 | 高（独立 instance） |

## 4. Agent Team — 多独立 session 协作（实验性）

多个独立 Claude Code 实例协调工作：
- 一个 lead 协调者 + 多个 teammate
- 共享 task list，teammate 自己认领
- **teammate 之间可以直接 peer-to-peer 消息**
- 每个 teammate 是完全独立的 session

### Agent Team vs Subagent

| | Subagent | Agent Team |
|--|----------|-----------|
| 通信方式 | 只回报 main agent | teammate 之间直接通信 |
| 协调方式 | main agent 管理所有工作 | 共享 task list + 自我协调 |
| 上下文 | 与主 session 隔离 | 完全独立 |
| 成本 | 低 | 高（多独立 instance） |
| 适用场景 | 专注任务，只关心结果 | 复杂需要讨论和协作的任务 |

### 关键架构：Mailbox Pattern（来自源码泄露）

teammate 执行危险操作时：
1. 不能自行批准
2. 向 coordinator mailbox 发请求
3. 等待审批/拒绝

这比"每个 agent 独立自治"要安全。

## 三层扩展的选型决策树

```
需要 LLM 判断吗？
  ├─ 否 → Hook（确定性脚本）
  └─ 是 → 需要并行独立工作吗？
            ├─ 否 → Skill（按需加载知识）
            └─ 是 → 需要 teammate 之间直接通信吗？
                      ├─ 否 → Subagent（隔离 + 结果汇报）
                      └─ 是 → Agent Team（独立 session + P2P 消息）
```

## 工程意义

1. **Hook 是最低成本的扩展**：不需要 LLM 的自动化应该用 Hook，不要用 Skill 浪费 token
2. **Skill 按需加载是关键**：解决了"文档太长了撑爆上下文"的问题——不调用不加载
3. **Subagent 和 Agent Team 是不同的并行模型**：前者适合隔离独立任务，后者适合需要协作的复杂问题
4. **teammate 直接通信是 Agent Team 的核心价值**：与 OpenClaw sessions_spawn 的"结果汇报给 main"模型完全不同
5. **OpenClaw 缺乏对等扩展机制**：OpenClaw 的 skill 是启动时加载快照，没有按需加载机制；sessions_spawn 也没有 P2P 通信
