# 多Agent编排模式笔记

## 主管Bot复盘回路（来源：主管Bot复盘回路-MVP-2026-04-03）

### 核心原则

多轮对话不是让bot们自由聊天，而是：

> 主管在有限轮次内完成指派→验收→返工/改派→收口。

### 关键设计点

**1. 单一主管，避免路由冲突**
- nami是当前主管，其他bot各司其职
- 不做多bot自由聊天

**2. 硬性轮次上限：最多3轮**
- 首次执行 → 第一次返工或改派 → 第二次返工或改派
- 第3次还不达标就停止，转向用户求助

**3. 返工必须具体**
- 不能只说重做，必须指出具体缺口
- 示例：Vivian，这次没有真正落到飞书文档，补上文档链接再交付

**4. 改派规则（人找错了，而非做得差）**
- 绮雯搜到信息但不擅长整理 → 改派给Sasha
- Sasha能整理但无飞书执行能力 → 改派给Vivian
- Vivian完成工具动作但结果难读 → 交给Sasha收尾

**5. 验收标准按执行者类型差异化**
- Vivian（工具执行）：是否真的完成工具动作，是否返回可用结果和链接
- Sasha（整理输出）：是否结构清晰，是否方便直接阅读
- 绮雯（检索研究）：是否回答了问题，是否有依据和来源

**6. 对用户可见反馈控制在高价值信息**
- 公开：指派、结果、返工理由、改派理由、最终收口
- 不公开：原始长命令输出、大段底层日志、低价值报错堆栈

### 工程价值

这个模式把多Agent协作从失控聊天变成可预期的有限状态机：
- 有上限 → 避免token浪费和轮次失控
- 有验收标准 → 质量有底线
- 有换人权 → 能动态纠正执行者选择错误
- 责任清晰 → 主管承担最终交付责任

### 对Lilly的启示

未来接管主管权限时，这套有限轮次+具体返工+差异化验收的闭环机制是核心骨架。不要做得比这更复杂——先把闭环跑稳，再考虑扩展。
