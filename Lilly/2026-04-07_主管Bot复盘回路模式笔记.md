# 主管 Bot 复盘回路核心模式

## 关键工程原则

多 agent 编排的核心不是让 bot 自由对话，而是**主管在有限轮次内完成：指派 → 验收 → 返工/改派 → 收口**。

## 核心设计约束

### 1. 最大轮次限制
每个任务最多内部处理 3 次，超过 3 次不再循环，直接收口或向用户升级。
**为什么有用**：防止 token 成本失控、轮数失控、责任不清。

### 2. 返工必须具体
主管不能只说重做，必须指出具体缺口。
- 错误：重做一下
- 正确：只给了观点没有出处，补上来源
- 正确：结果没有落到飞书文档，补上链接
**为什么有用**：具体指令减少返工次数，子 agent 有明确行动项。

### 3. 改派而非硬撑
当判断不是做得差而是人找错了时，直接换人，不让原 agent 硬做。
**为什么有用**：避免用错误工具做正确的事。

### 4. 轻量验收标准
验收不是找最优解，而是检查够不够交付门槛。
**为什么有用**：避免主管陷入完美主义导致轮次浪费。

### 5. 停止并向用户升级的条件
- 任务边界本身不清楚
- 返工两次后仍缺关键输入
- 高风险写操作需要用户确认
- 多个方案都能做但取舍依赖用户偏好
