# 软件形态、包、库、框架、依赖、运行时、包管理器 笔记

## 0. 总览

理解软件，先别按“看起来像什么”分，先按这三层分：

1. **它本质上是什么角色**
   - 应用程序
   - 命令行工具
   - 后台服务
   - 库
   - 框架
   - 运行时
   - 编译器
   - 插件

2. **它怎么被分发和安装**
   - `.exe` / `.msi`
   - `.deb` / `.rpm`
   - `.zip` / `.tar.gz`
   - `npm` 包
   - `pip` 包
   - Docker 镜像

3. **它和别的软件是什么关系**
   - 独立可运行
   - 依赖某个运行时
   - 被别人调用
   - 需要额外二进制
   - 需要系统库 / 动态库

很多混乱都来自把这三层混成一层。

---

## 1. 软件最常见的角色

### 1.1 应用程序（Application）

面向最终用户或操作者，自己就能独立跑起来。

#### GUI 应用
有图形界面，有窗口。

例子：
- 微信
- Edge
- VS Code
- Photoshop

#### CLI 应用
没有窗口，靠命令行使用。

例子：
- `git`
- `gcc`
- `python`
- `node`
- `ffmpeg`

#### 后台服务 / Daemon
常驻后台运行，不一定给人直接点开。

例子：
- `nginx`
- `mysqld`
- `redis-server`
- 某些 MCP server
- 某些本地代理程序

---

### 1.2 库（Library）

库不是“给人直接点开用”的，它是**给别的软件调用的能力集合**。

特征：

- 通常不作为最终用户入口
- 主要被代码 `import` / `require` / `link`
- 重点是提供 API，不是提供完整工作流

例子：
- Python 的 `requests`
- Node.js 的 `react`
- Playwright 作为库时的 API
- C/C++ 的 `.lib` / `.a` / `.so` / `.dll`

一句话：

> 库像工具箱，程序自己决定什么时候拿工具。

---

### 1.3 框架（Framework）

框架比库更“大”，它不只是给你 API，还给你**结构、约定、生命周期、入口**。

特征：

- 规定项目组织方式
- 提供统一运行机制
- 约束比库更强
- 你是在它的骨架里填代码

例子：
- Spring
- Django
- Laravel
- `@playwright/test`

一句话：

> 库是你调用它；框架是你把代码交给它来组织运行。

#### 和 Spring 的类比

这个类比成立。

- **Spring**：Java 应用框架
- **Playwright Test**：浏览器自动化测试框架

它们用途不同，但“框架”这个抽象层次相同：都不是单一函数库，而是一套组织机制。

---

### 1.4 插件（Plugin / Extension）

插件不能独立运行，必须挂在宿主程序里。

例子：
- VS Code 扩展
- 浏览器扩展
- Photoshop 插件
- 某些 JetBrains 插件

一句话：

> 插件是“寄生型软件”。

---

### 1.5 运行时（Runtime）

运行时代替某种语言或字节码提供执行环境。

例子：
- Python 解释器
- Node.js
- JVM
- .NET Runtime

运行时不是业务功能本身，但很多程序离开它就跑不起来。

例如：

- `.py` 代码离不开 Python
- Java 程序离不开 JVM
- 很多 JS 工具离不开 Node.js

一句话：

> 运行时是“软件赖以活着的环境”。

---

### 1.6 编译器（Compiler）与工具链（Toolchain）

#### 编译器
把源代码翻译成机器码或别的中间产物。

例子：
- `gcc`
- `clang`
- `javac`
- `tsc`

#### 工具链
不只一个编译器，还包含链接器、汇编器、标准库、头文件、调试工具等。

例子：
- GCC toolchain
- Android NDK
- Rust toolchain

一句话：

> `gcc` 不只是“一个包”，它本质上是编译工具系统的一部分。

---

## 2. 软件“长什么样”和“是什么”不是一回事

### 2.1 `.exe` 只是文件格式，不是软件类别

Windows 下常见的 `.exe` 只是**可执行文件格式**。

它可以是：

- GUI 应用
- CLI 工具
- 后台服务
- 安装器
- 启动器

所以：

> `.exe` 不是“软件类型”，只是 Windows 上一种装载和执行形式。

#### 例子
- `python.exe`：CLI 程序
- `msedge.exe`：GUI 程序
- 某安装器 `setup.exe`：安装程序，不是最终应用本体

---

### 2.2 安装包也不是程序本体

安装包是用来“把程序装到系统里”的。

常见安装包：
- Windows：`.exe`、`.msi`
- macOS：`.pkg`
- Linux：`.deb`、`.rpm`

安装包里可能包含：

- 可执行文件
- 动态库
- 配置文件
- 快捷方式
- 注册表项
- 卸载信息

所以：

> 安装包是运输工具，不等于程序本体。

---

### 2.3 压缩包 / 便携版

有些软件不走安装器，直接压缩包分发：

- `.zip`
- `.7z`
- `.tar.gz`

解压后就能运行。

这种软件常见于：
- 便携工具
- CLI 工具
- 开发工具
- 跨平台二进制分发

---

## 3. “包（package）”到底是什么

包不是功能分类，包是**分发单位**。

一个包通常包含：

- 代码
- 版本信息
- 元数据
- 依赖声明
- 可执行入口
- 安装说明

包里面可能装的是：

- 一个库
- 一个框架
- 一个 CLI 工具
- 一个插件
- 一组配置模板
- 一套资源文件

所以：

> 包更像物流箱子，不是“内容本身”。

---

## 4. 常见的分发形式

### 4.1 操作系统层面的包

#### Windows
- `.exe`
- `.msi`

#### Linux
- `.deb`
- `.rpm`

#### macOS
- `.pkg`
- `.app`

---

### 4.2 语言生态层面的包

#### Node.js
- npm 包

#### Python
- pip 安装的包
- wheel / sdist

#### Java
- JAR
- Maven / Gradle 坐标

#### Rust
- crate

#### Go
- module

这些“语言包”本质上仍然是包，只是它们由某个语言生态的工具来管理。

---

### 4.3 容器镜像

如 Docker image。

它不是普通安装包，而是：

- 程序
- 依赖
- 运行环境
- 文件系统状态

一起打包。

一句话：

> 镜像是“整个运行环境”的分发形式。

## 5. 依赖是什么

依赖不是文件类型，也不是软件类别。

依赖是：

> A 要运行，必须依赖 B。

#### 依赖可能是什么
- 一个库
- 一个运行时
- 一个外部程序
- 一个系统动态库
- 一个数据库服务
- 一个浏览器二进制
- 一个配置文件
- 一个驱动

#### 例子
- Python 脚本依赖 Python 运行时
- Java 程序依赖 JVM
- Playwright 依赖浏览器二进制
- 某些 Windows 程序依赖 `vcruntime*.dll`
- Web 应用依赖数据库服务

所以：

> 依赖描述的是“关系”，不是“长相”。

---

## 6. 静态库、动态库、共享库

### 6.1 静态库

在编译/链接时被直接并进最终程序。

特点：
- 最终产物更大
- 运行时不需要外部同名库
- 部署更简单

常见形式：
- `.a`
- `.lib`

---

### 6.2 动态库 / 共享库

运行时再加载。

特点：
- 主程序更小
- 多个程序可共享
- 部署时要确保库存在
- 版本不兼容时容易出错

常见形式：
- Windows：`.dll`
- Linux：`.so`
- macOS：`.dylib`

一句话：

> 静态库是编进去，动态库是运行时再连上去。

---

## 7. 语言包和系统程序的区别

### 7.1 `gcc` / `python` 属于什么

它们首先是**可执行程序**：

- `gcc`：CLI 编译器工具
- `python`：CLI 解释器/运行时

它们可能通过：
- 安装器
- 系统包管理器
- 压缩包
- Homebrew / winget / apt

来安装。

但无论怎么装，它们本质仍是**程序**。

---

### 7.2 `npm install react` 这种装的是什么

装的是一个 **Node 包**。

这个包的内容可能是：
- 库
- 框架
- CLI
- 类型定义
- 工具链的一部分

所以：

> `npm/pip/pnpm` 装的是“包”，不是固定只装“库”。

---

## 8. 包管理器和命令执行器

### 8.1 包管理器（Package Manager）

包管理器负责：

- 下载包
- 安装包
- 解析依赖
- 锁定版本
- 更新包
- 删除包

#### 例子

##### Node.js 生态
- `npm`
- `pnpm`
- `yarn`

##### Python 生态
- `pip`

##### Linux 系统级
- `apt`
- `yum`
- `dnf`

---

### 8.2 命令执行器

命令执行器重点不是“安装”，而是“执行”。

#### `npx`
`npx` 不是包管理器。

它的作用是：
- 运行某个 npm 包暴露出来的命令
- 优先用当前项目本地已安装的包
- 必要时临时拉一个包来执行

例如：

```bash
npx playwright test
```

含义是：

执行 `playwright` 这个包提供的 CLI 命令。

不是：

把 Playwright 永久导入项目。

所以一定区分：

- `npm/pnpm`：装包
- `npx`：跑命令

---

## 9. npm、pnpm、pip、npx 之间的关系

### 9.1 npm
Node.js 默认包管理器。

作用：
- 安装 Node 包
- 管理 `package.json`
- 生成 `node_modules`

---

### 9.2 pnpm
也是 Node.js 包管理器。

和 npm 同类，但特点不同：
- 更省磁盘
- 安装更快
- 依赖复用更强
- monorepo 体验更好

---

### 9.3 pip
Python 包管理器。

装的是 Python 生态里的包，不是 Node 包。

---

### 9.4 npx
不是包管理器，是执行器。

#### 记忆法
- `npm` / `pnpm` / `pip`：负责“拿货、装货、管货”
- `npx`：负责“把某个命令跑起来”

---

## 10. 测试框架是什么

测试框架不是“某几个断言函数”，而是：

> 围绕测试活动的一整套组织系统。

它通常包含：

- 测试用例定义方式
- 测试发现机制
- setup / teardown
- fixture
- 断言
- 并行执行
- 隔离环境
- 报告
- 调试工具

### 例子
- JUnit
- pytest
- Jest
- Playwright Test

### 和普通库的区别
普通库只给你工具函数。  
测试框架给你的是：

- 怎么写
- 怎么跑
- 怎么组织
- 怎么看结果

---

## 11. Playwright 放回这套分类里怎么理解

Playwright 这个词容易混，是因为它在不同上下文里指不同东西。

### 11.1 作为库
它是浏览器自动化库。

提供能力：
- 启动浏览器
- 打开页面
- 点击
- 输入
- 截图

---

### 11.2 作为测试框架
`@playwright/test` 是测试框架。

它在“浏览器控制”之上又加了：
- test runner
- 断言
- fixture
- trace
- report

---

### 11.3 作为 CLI
`npx playwright ...` 是命令行入口。

可以：
- 跑测试
- 安装浏览器
- 生成代码
- 看 trace

---

### 11.4 作为 MCP
`@playwright/mcp` 是 MCP server 形态。

它把浏览器操作能力通过 MCP 协议暴露给 AI。

所以：

> Playwright 不是只有一种形态。

---

## 12. Chromium 和 Playwright 的关系

Chromium 不是 Playwright 本体。

关系是：

- Playwright：控制层
- Chromium：被控制的浏览器

Playwright 可以控制：
- Chromium
- Firefox
- WebKit

所以：

> Playwright 像遥控器，Chromium 像电视。

---

## 13. Playwright “安装 Chromium” 到底是什么意思

这里有两层安装：

### 13.1 安装 Node 包
例如：

```bash
npm install -D @playwright/test
```

这一步装的是：
- JS/TS 代码
- 测试框架
- CLI
- 依赖元数据

---

### 13.2 安装浏览器二进制
例如：

```bash
npx playwright install chromium
```

这一步装的是：
- 真正的 Chromium 浏览器程序

所以：

> 不是 npm 装了 Playwright，就等于 Chromium 已经有了。

更准确的说法是：

> Playwright 依赖浏览器二进制运行；Playwright CLI 可以帮你把这些浏览器二进制下载到本地。

## 14. 软件能不能没有客户端

可以。

“客户端”通常指给人交互的前端程序，但很多软件根本不需要 GUI 客户端。

### 没有图形客户端的软件
- `gcc`
- `python`
- `git`
- `ffmpeg`
- `redis-server`
- `nginx`

它们仍然是完整软件，只是入口不是窗口，而是：
- CLI
- API
- socket
- 配置文件
- 网络协议

所以：

> “有客户端”不是软件成立的必要条件。

## 16. 最短记忆版

### 关于软件
- 软件不是只有两种。
- “程序 / 库 / 框架 / 运行时 / 插件”是角色分类。

### 关于文件和安装
- `.exe` 只是 Windows 可执行形式，不是软件类别。
- 安装包不等于程序本体。
- 包是分发单位，不是功能类别。

### 关于依赖
- 依赖是软件之间的关系。
- 依赖不一定是库，也可能是运行时、服务、浏览器、动态库。

### 关于包管理器
- `npm` / `pnpm` / `pip` 是包管理器。
- `npx` 不是包管理器，是命令执行器。

### 关于框架
- 库像工具箱。
- 框架像已经搭好的施工骨架。
- 测试框架就是专门组织“写测试、跑测试、看结果”的框架。

### 关于 Playwright
- 可以是库
- 可以是测试框架
- 可以通过 CLI 使用
- 也可以以 MCP server 形式工作
- 它不是 Chromium，但能控制 Chromium

### 关于 Chromium
- Chromium 是浏览器程序
- Playwright 控制它
- Playwright CLI 可以帮你下载安装它
