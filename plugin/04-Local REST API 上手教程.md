# Local REST API 上手教程

## 这个插件到底是干什么的

`Local REST API` 的本质是：

- 在你本机启动一个 REST API 服务
- 让外部程序能读写你的 Obsidian 笔记

也就是说，它不是“笔记插件”，而是“自动化接口插件”。

## 原理

插件启动后，会在本地监听一个端口。

外部工具可以通过 HTTP 或 HTTPS 请求：

- 读取笔记
- 搜索内容
- 修改文件
- 创建文件
- 获取 vault 信息

所以它的核心价值是：

- 把 Obsidian 变成可编程的本地服务

## 你该怎么理解它

### 适合

- 你要用脚本批量处理笔记
- 你要让外部工具接入 Obsidian
- 你要做本地自动化

### 不适合

- 只是普通记笔记
- 没有明确自动化需求
- 对本地服务和安全概念不熟

## 配置页面逐项解释

### `Enable Non-encrypted (HTTP) Server`

作用：

- 开启一个不加密的 HTTP 服务

风险：

- 明文传输
- 更容易被误暴露

推荐值：

- 关

### `Show advanced settings`

作用：

- 展开高级设置

推荐：

- 只有你明确知道自己在改什么时再开

### `Enable Encrypted (HTTPS) Server`

作用：

- 开启加密的本地 HTTPS 服务

推荐值：

- 开

### `Encrypted (HTTPS) Server Port`

作用：

- 设置 HTTPS 服务监听端口

你当前值：

- `27124`

建议：

- 没特殊需求就别改

### `Non-encrypted (HTTP) Server Port`

作用：

- 设置 HTTP 服务端口

建议：

- 因为 HTTP 本来就不建议开，所以这个通常不用动

### `API Key`

作用：

- 所有调用方访问 API 时使用的认证凭据

原则：

- 这是密钥，不是普通配置
- 不要贴到公开仓库
- 不要截图外发

### `Certificate Hostnames`

作用：

- 证书允许的主机名

适合谁改：

- 真正理解本地 HTTPS 证书的人

普通用法：

- 不动

### `Certificate / Public Key / Private Key`

作用：

- 本地 HTTPS 的证书和密钥材料

结论：

- 没有明确原因，不要手改

### `Authorization Header`

作用：

- 指定 API 请求里用哪个请求头携带认证信息

适合谁改：

- 你在写接入脚本，且需要兼容特殊客户端

普通用法：

- 默认即可

### `Binding Host`

作用：

- 指定服务绑定到哪个主机地址

风险：

- 配不对可能把本地服务暴露到更大范围

建议：

- 保持默认本地绑定

### `Reset All Cryptography`

作用：

- 重置证书、密钥等加密材料

什么时候用：

- 你确实怀疑密钥泄露
- 你想彻底重建本地 HTTPS 配置

### `Re-generate Certificates`

作用：

- 重新生成证书

### `Restore Default Settings`

作用：

- 恢复插件默认设置

## 推荐你怎么配

如果你只是“装着备用”，最稳配置就是：

- HTTPS：开
- HTTP：关
- 端口：默认
- API Key：保留，但不要外泄
- 高级设置：别乱改

## 风险边界

这个插件最核心的风险不是“它会不会崩”，而是：

- 一旦配置过宽，外部程序就能改你的笔记

所以底线是：

- 不把它暴露到公网
- 不随便开启 HTTP
- 不公开 API Key

把 `Local REST API` 当成“自动化接口层”，不是普通设置插件。  
没明确需求时，知道它存在就够了。
