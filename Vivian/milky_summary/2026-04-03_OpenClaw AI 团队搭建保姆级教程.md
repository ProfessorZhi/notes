# OpenClaw AI 团队搭建保姆级教程

## 什么是 OpenClaw

OpenClaw 是一个多智能体（Multi-Agent）协作框架，允许用户搭建由多个 AI Agent 组成的团队，实现"一个人 = 一个团队"的工作模式。每个 Agent 可以像员工一样各司其职，可以有定时任务、24小时不间断工作，或按需触发。

---

## OpenClaw 三层架构（核心原理）

理解 OpenClaw 的工作原理是搭建团队的基础，整个系统分为三层：

### 第一层：全局层（Global）

所有 Agent 共用的公共设施，**不是每个 Agent 独有的**。

核心文件：`opencloud.json`（总配置文件），包含以下参数：

```json
// opencloud.json 主要配置项
{
  "model_provider": "大模型厂商定义",
  "identity": "登录身份",
  "agents": "agent定义有哪些",
  "default_model": "默认模型",
  "workspace": "工作目录",
  "tools": "全局工具权限",
  "calling_rules": "agent之间调用规则",
  "browser": "浏览器信息",
  "telegram": "telegram外部渠道连接",
  "gateway": "网关",
  "memory": "记忆配置"
}
```

> ⚠️ **重要提示**：大部分崩溃或对话无反应的问题，都源于误改 `opencloud.json` 文件。修改前务必谨慎，可以让 OpenClaw 先备份并通知你。

### 第二层：Agent 层（每个 Agent 独立领地）

每个 Agent 都有自己独立的工作空间、对话记录。Agent 层的文件在**每个对话开始时被加载**作为上下文。

| 文件名 | 作用 |
|--------|------|
| `agent.md` | Agent 工作说明书，定义做事规则、优先级、边界、如何使用记忆 |
| `so.md` | Agent 性格和说话方式 |
| `user.md` | 用户理解，用户称呼偏好、注意事项 |
| `identity.md` | Agent 身份卡，定义叫什么、是谁 |
| `skills.md` | 工具说明，本地有哪些工具、命令约定 |
| `memory.md` | 长期精华记忆（非流水账，是提炼后的信息） |
| `*.md`（同名文件） | 若与全局同名，Agent 层优先级更高，会覆盖全局配置 |

**关键点**：同一个 Agent 不同的对话线程之间**共享**这套文件，但不同 Agent 之间**不共享**。

### 第三层：Session 层（对话线程）

这是 Agent "当前在哪个窗口工作"的层级。

- 同一 Agent 在不同 Discord 频道/Telegram 对话 = 不同线程
- 命令 `new` 或 `slash` 会开启新线程
- 区别：`slash new` 会清空上下文重新开始，旧对话仍在

**核心文件**：每个 Agent 下的 `sessions/` 文件夹存放各对话线程的历史记录。

## UP主的 6 人 AI 团队架构

### Agent 1：情报员（攻击）

| 属性 | 说明 |
|------|------|
| 触发方式 | 每日定时（每天早上9点自动执行） |
| 主要功能 | 搜索 AI/Agent 相关技术进展、产品进展、用户痛点 |
| 工作方式 | 搜索信息后汇报 → 若不回复则明天继续；若回复要求深挖，则针对特定话题继续搜索 |
| 使用场景 | 保持对行业动态的持续跟踪 |

### Agent 2：私人助理/提醒助手

| 属性 | 说明 |
|------|------|
| 触发方式 | 随时对话 + 每日定时（每天早上7:30） |
| 主要功能 | 记录待办事项、事情进展；回答问题；天气播报 |
| 工作方式 | 接收语音/文字输入 → 记录 → 按重要性排序提醒 |
| 使用场景 | 完全替代笔记软件（如 Notion） |

### Agent 3：主要助手（Max 狮子）⭐最重要

| 属性 | 说明 |
|------|------|
| 权限 | 最高（最好的模型） |
| 主要功能 | 日常重要对话/长期计划讨论；视频流水线 |
| 视频流水线流程 | 定时搜索 → 筛选热门帖子 → 翻译+收集数据 → 用户点赞 → 研究背景信息 → 用户再点赞 → 生成封面建议/脚本大纲 |
| 辅助功能 | 收藏夹功能（对有用的帖子点眼睛保存） |
| 监控平台 | X (Twitter)、YouTube、Hacker News |

### Agent 4：开发者（潘大熊猫）

| 属性 | 说明 |
|------|------|
| 触发方式 | 几乎 24 小时不间断 |
| 主要功能 | 开发任务 + 无任务时自动测试/找 bug/边缘情况测试 |
| 适用场景 | 高价值工作（开发、销售线索挖掘、产品推广） |

> 💡 **24小时运转的成本考量**：
> - 云端好模型：成本很高
> - 本地模型（Max Studio）：硬件成本高
> - **建议**：24小时任务必须是高价值工作

### Agent 5：内容转化（monkey 猴子）

| 属性 | 说明 |
|------|------|
| 触发方式 | 被动触发（给链接才工作） |
| 主要功能 | 将 YouTube 视频链接转换为带截图的文章 |
| 使用场景 | 视频内容二次利用/多平台分发 |

### Agent 6：安全与更新（tiger 老虎）

| 属性 | 说明 |
|------|------|
| 触发方式 | 每日定时 |
| 主要功能 | 扫描安全问题；通知可更新版本；评估更新风险（是否影响当前任务） |

## 如何创建新 Agent

### 方法：通过主要 Agent 智能创建（推荐）

**步骤 1**：在主 Agent 对话中发送指令

```
"新建一个 agent，他叫什么，他要做什么，有什么样的任务，最后告诉他你的 [bot token]"
```

**步骤 2**：获取 Bot Token

1. 打开 BotFather（Telegram 官方机器人）
2. 点击 `Create new bot`
3. 输入 bot 名称
4. 获得 Token，复制

**步骤 3**：将 Token 发送给主要 Agent，它会自动完成创建

### 重要文件存放位置

- **全局配置**：`opencloud.json`（所有 Agent 共用）
- **单个 Agent 配置**：对应 Agent 文件夹下
- **对话记录**：`sessions/` 文件夹

## 多 Agent 协作方式

### 方式一：固定分工（长期团队）

类似真实公司组织架构，多个独立 Agent 各司其职，长期承担固定职责。

**协作触发示例**：

```
Agent A（情报员）搜到热门帖子 
→ 用户在 Discord 点"大拇指"表示感兴趣
→ Agent B（Max）自动开始研究背景信息
→ 用户再点"大拇指"
→ Agent C 开始制作封面/脚本
```

**适用场景**：日常工作流分工，长期稳定任务

### 方式二：临时 sub-agent（子代理）

使用 `sessions spawn` 命令创建隔离的子代理，在独立 Session 中工作。

**特点**：
- 子代理在独立线程工作，不阻塞主对话
- 主 Agent 可以继续回答其他问题
- 完成后主动汇报结果到主频道

**命令示例**：

```
用 sessions spawn 研究 open claw 多智能体最近最受欢迎的三个讨论点，
要求每个讨论点的热度、原因、代表观点，以及我可以做视频用的切入角度。
让它在后台启动，完成后主动汇报。
```

**适用场景**：
- 研究类慢任务
- 重任务
- 并行任务
- 不会阻塞主对话

## 何时需要/不需要多个 Agent

### ✅ 适合用多个 Agent 的场景

1. **任务天然可拆分**：研究 → 写作 → 审核 → 发布（本来就像不同角色）
2. **需要不同工作区**：不同 Agent 有不同文件，边界清楚
3. **任务可并行**：查资料和整理结构可以同时进行
4. **避免污染**：单 Agent 做太多不同类型任务会互相干扰，长期记忆会把不该混在一起的东西混合
5. **重复性/重任务**：适合并行处理

### ❌ 不需要多个 Agent 的场景

- 任务很短
- 上下文高度连续，不想打破
- 角色边界拆不清
- 为了"看起来高级"而硬拆
- 一个 Agent 能做好的事

> 💡 **核心原则**：像一个队伍一样，当任务需要一个团队时才用团队。

## 实践建议

### 1. 从少到多，逐步添加

- 不要一上来创建很多 Agent
- 从第一个主要的 Agent（默认的）开始
- 例如：先加一个"每日信息研究"Agent，跑顺后再根据业务需求添加下一个
- **每加一个都要先跑顺功能，验收效果**

### 2. 控制工作区文件长度

特别需要精简的文件：
- `so.md`（性格描述）
- `user.md`（用户理解）
- `heartbeat.md`（心跳文件）

**原因**：这些文件都会进入 Agent 运行上下文，如果太长：
- 效果降低（上下文稀释）
- **成本成倍增加**（多个 Agent 都加载大量文件时）

### 3. 善用备份机制

让 AI Agent 在修改重要配置前：
1. 先备份
2. 测试效果
3. 获得你的明确许可后再修改

## 弹幕精选有价值的问题

| 问题 | 说明 |
|------|------|
| **token 成本问题** | 云端模型 24 小时运转确实贵；本地部署（如 Ollama）可降低成本 |
| **与普通 AI 聊天区别** | 核心区别在于：多 Agent 有独立工作区、长期记忆、定时任务、协作流程 |
| **任务污染** | 单 Agent 做太多不同任务确实会污染上下文，多 Agent 可以隔离 |
| **隐私问题** | 重要数据打包发给 API 云厂商是事实，需自行评估风险 |
| **与 Ollama 对比** | OpenClaw 是多 Agent 框架，Ollama 是本地模型运行平台，功能定位不同 |

---

## 总结

OpenClaw 的核心价值在于：

1. **角色分离**：不同 Agent 有不同工作区、性格、能力
2. **自动化流程**：定时任务 + 触发式协作
3. **并行工作**：sub-agent 不阻塞主对话
4. **长期记忆**：提炼后的记忆文件在各对话间共享

搭建 AI 团队的关键是：**从实际需求出发，从少到多，让每个 Agent 先跑顺再扩展**。
