# Agent 持续学习的三层框架

## 核心观点

Agent 的持续学习可以发生在**三个层面**，并非只有更新模型权重这一条路径：

| 层级 | 学习内容 | 典型方式 |
|------|----------|----------|
| **Model** | 模型能力本身 | 微调、强化学习 |
| **Harness** | 执行框架和流程 | Traces + Evals + Code Iteration |
| **Context** | 运行时配置和记忆 | 离线或在线更新配置 |

> 来源：LangChain 创始人 Harrison Chase 的文章 [Continual Learning for AI agents](https://x.com/hwchase17/status/2040467997022884194)

---

## 三层定义详解

### 第一层：Model（模型层）

**定义**：模型权重本身的学习，即传统意义上的模型训练/微调。

**学习内容**：
- 推理模式
- 任务能力
- 行为倾向

**优势**：
- 影响上限最高
- 效果全局性、长期性

**问题**：
- 成本高
- 周期长
- 可检查性差
- **灾难性遗忘**：新数据写入后，旧能力可能被覆盖
- 粒度粗，难以按用户/租户分别更新

---

### 第二层：Harness（执行框架层）

**定义**：包裹在模型外部的执行框架，包括代码、提示词组织、工具调用逻辑、工作流编排。

**学习机制**：更像工程闭环

```
1. 跑一批任务，产生 Execution Traces
2. 对结果做评估
3. 把日志和轨迹存下来
4. 让 Coding Agent 分析 Traces
5. 提出 Harness Code 的修改建议
```

**更新内容**：
- 提示词的组织方式
- 工具调用逻辑
- 任务分解策略
- 异常处理流程
- 失败后的回退方式
- 整个 Agent 循环

---

### 第三层：Context（运行时配置层）

**定义**：运行时加载的外部配置，包括指令、技能、工具、记忆。

**更新内容**：
- 指令（Instructions）
- 技能（Skills）
- 工具（Tools）
- 记忆（Memory）

**关键区分**：同样是指令/技能/工具，关键在于它属于**系统固定框架**还是**运行时配置**。

**记忆更新的两种方式**：
| 方式 | 说明 |
|------|------|
| 显式 | 用户主动要求系统记住 |
| 隐式 | 系统根据规则自动决定什么值得沉淀 |

**学习方式的灵活性**：
- 离线分析 Traces → 批量更新 Context（任务结束后的经验整理）
- 执行过程中直接更新 Memory（边做边学）

**粒度优势**：可以按 Agent、用户、团队、租户分别维护不同的 Context

### Claude Code 架构

| 层级 | 组件 |
|------|------|
| Model | 底层模型 |

| 层级 | 组件 |
|------|------|
| Model | 模型 |
| Harness | 派加的脚手架、执行框架 |
| Context | Agent 自己的 Memory 和 Skills |

## 三层完整对比

| 维度 | Model | Harness | Context |
|------|-------|---------|---------|
| **形态** | 权重文件 | 代码 | 配置文件 |
| **更新成本** | 最贵 | 中等 | 最轻 |
| **更新速度** | 最慢 | 中等 | 最快 |
| **可检查性** | 最难 | 较强 | 最容易 |
| **影响上限** | 最高 | 中等 | 低于模型层 |
| **粒度** | Agent 级 | Agent 级 | 可细化到用户/团队/租户 |
| **更新方式** | 批量离线 | 批量离线 | 离线或运行中在线更新 |
| **学习内容** | 通用能力 | 执行机制 | 偏好、经验、规则、组织知识 |

## 决策框架

面对不同问题，应该动哪一层？

| 问题类型 | 推荐层级 | 原因 |
|----------|----------|------|
| 长期底层通用能力不足 | **Model** | 需要从根本上提升模型能力 |
| 执行稳定性、工具使用、流程编排问题 | **Harness** | 改变系统使用模型的方式 |
| 偏好、记忆、组织知识、角色差异 | **Context** | 运行时信息更新，灵活高效 |

## 企业环境的额外约束

Box CEO Aaron Levi's 重要补充：**权限边界**。

核心问题不只是"技术上能不能学到"，还包括：**知识进入模型后，还能不能按角色/团队/租户精细切开**。

| 问题 | Model 层 | Context 层 |
|------|----------|------------|
| 权限切分 | 知识沉浸后难以按权限切分 | 天然支持按用户/团队单独维护 |
| 可解释性 | 难以解释输出受哪些内部信息影响 | 更易与权限体系绑定 |

**结论**：企业里的很多持续学习能力更适合落在 Context 层。

## 核心洞察：Execution Traces 是共同的学习原料

无论更新哪一层，**Execution Traces** 是三层持续学习共同依赖的基础材料：

```
Execution Traces
    │
    ├──→ Model 层：作为训练材料
    ├──→ Harness 层：定位问题，改代码，改流程
    └──→ Context 层：抽取经验，更新记忆和配置
```

**Trace 的地位**：
- 不只是日志
- 不只是观测数据
- 不只是调试工具
- **是 Agent Learning 的基础设施**

**可观测性不只是排障工具，它本身就是 Agent Learning 的基础设施。**

---

## 总结

Agent 要持续变好的关键：**不是只换更强的模型，而是想清楚该让哪一层学习。**

| 层级 | 学什么 | 代价 | 上限 |
|------|--------|------|------|
| Model | 能力 | 最高 | 最高 |
| Harness | 如何执行 | 中等 | 中等 |
| Context | 经验与配置 | 最低 | 受限于模型能力 |

三层并非互斥，而是可以根据问题类型灵活选择或组合使用。
