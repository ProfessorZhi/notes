# Claude Code Team 模式完全指南

## 什么是 Team 模式

Claude Code 的 Team 模式是一种多智能体协作方案，通过启动多个子智能体（Sub-Agents）并行或分工处理任务，这些子智能体持续待命，不会在每次调用时重置上下文。

> 对比：旧的 SAG（Single Agent Group）模式每次启动时上下文会完全重置。

### Team 模式 vs SAG 模式核心区别

| 特性 | SAG 模式 | Team 模式 |
|------|---------|-----------|
| 上下文持久性 | 每次启动重置 | 子智能体持续待命 |
| 上下文管理 | 200K 上下文容易溢出 | 任务拆分到各子智能体，分摊上下文压力 |
| 性能衰减 | 压缩上下文后性能急剧下降 | 上下文分配到各子 Agent，整体稳定 |
| Token 消耗 | 基准水平 | 约为 SAG 的 2 倍及以上 |

## Team 模式的核心优势

### 1. 上下文不再容易溢出

使用 SAG 模式时，200K 上下文的主智能体在执行任务时，大约只能进行 5~10 轮对话就会触及上下文上限。此时必须压缩上下文，性能会急剧下降。

Team 模式通过将任务拆分到多个子智能体，每个子智能体各自维护独立的上下文，**虽然总 Token 消耗增加了，但没有一个智能体的上下文被突破**，整体性能保持稳定。

### 2. 任务自动模块化拆分

子智能体在交互时存在信息损失问题，因此必须将任务拆分为小模块。配置好提示词后，Team 模式会：

- 主智能体（Leader）自动将任务拆分
- 分配给对应的子智能体
- 各子智能体独立执行并返回结果

## 推荐的 6 子智能体团队配置

项目已开源，包含以下 6 个子智能体角色：

| 角色 | 职责 | 文档数量 |
|------|------|---------|
| **后端开发** (Backend Dev) | 后端功能开发 | 3 个文档 |
| **前端开发** (Frontend Dev) | 前端界面开发 | 3 个文档 |
| **研究员** (Researcher) | 需求调研、技术调研 | 3 个文档 |
| **端到端测试** (E2E Tester) | 测试用例编写与执行 | - |
| **审查员** (Reviewer) | 代码审查 | - |
| **重构清洁器** (Refactorer) | 代码重构优化 | - |

> 项目灵感来源：`Ding` 项目、Claude Code 黑客松冠军分享的子智能体配置、`dada` 项目（非常火的 skills）。

## 安装与配置方法

### 安装步骤

1. 项目中有一个 `skills` 目录
2. 将该 skills 安装到 Claude Code 中

### ⚠️ 关键配置要求

**必须让 Opus 模型全量读取 skills：**

```
禁止行为：让探索型智能体去读取 skills
```

探索型智能体如果没有完整了解 skills 的全部内容就去创建项目，会导致创建结果不理想。

**正确做法：** 直接让 Opus 全量读取整个 skills，确保它理解每个子智能体的完整功能和配置。

## 任务启动流程

1. 配置好 skills 后，Opus 会主动询问：
   - "你想要完成一个什么样的任务？"
   - "需要哪些子智能体参与？"

2. Opus 会说明每个子智能体的功能

3. 用户确认后，Opus 自动创建对应的子智能体团队

## 文件持久化系统（重要特性）

Team 模式将所有进度、Plan 和发现写入文档，而不是存放在子智能体自己的全局状态中。这带来两个好处：

1. **透明可见**：用户可以实时查看每个角色的进展
2. **独立追踪**：每个角色维护自己的文档集合

### 文档结构

```
项目目录/
├── leader/                    # 主智能体文档
│   ├── progress.md
│   ├── plan.md
│   └── findings.md
├── backend/                   # 后端开发文档
│   ├── progress.md
│   ├── plan.md
│   └── findings.md
├── frontend/                  # 前端开发文档
│   ├── ...
├── researcher/                # 研究员文档
│   ├── ...
├── [项目子文件夹]/            # 每个具体任务创建独立子文件夹
│   ├── [任务1]/               # 任务级文档
│   └── [任务2]/
```

## 实际使用效果

### 能做到
- Leader 自动分发任务给子智能体
- 完成小功能开发 + 测试执行
- 200K 上下文的任务被分摊到各子 Agent，每个可能分到 100K

### 不能做到
- 完全自动化开发（人工仍需介入测试）
- 特别适合的场景：**稍微大型一点的项目**，上下文会突破 200K 的场景

### 典型表现
- 随便一个任务几乎都会突破 200K 上下文
- Team 模式将这个压力分摊给多个子 Agent，性能更稳定

## 成本与资源考量

### Token 消耗
- Team 模式的 Token 消耗约为普通模式的 **2 倍及以上**
- 如果项目较小（200K 上下文足够），使用 Team 模式反而浪费

### 成本优化方案

作者团队的实际做法：
- 在国外云服务器部署 Claude Code
- 多人拼团使用 **20X** 级别的账号
- 每人日均消耗 **10X** Token

### 使用建议

| 场景 | 推荐 |
|------|------|
| 小型项目，200K 上下文足够 | 不需要使用 Team 模式 |
| 中大型项目，上下文经常溢出 | **强烈推荐使用 Team 模式** |
| Token 预算有限 | 谨慎评估后再使用 |
| 需要高效率开发多人协作 | Team 模式性价比高 |

## 总结

Claude Code Team 模式的核心价值在于**将超出上下文的复杂任务拆解到多个子智能体中**，虽然 Token 消耗翻倍，但换来了上下文管理和性能的巨大提升。配合文件持久化系统，用户可以全程可视化每个角色的工作进度。对于中大型项目开发，这是目前性价比最高的多智能体协作方案。
