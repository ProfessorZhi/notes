# Claude Code 内部 9 类 Skill 能力地图

## 核心概念：什么是 Skill？

Skill 是一组**可复用能力资源**，包含：

- 脚本
- 模板
- 参考资料
- 数据文件
- 钩子（Hook）

Claude Code 可以读取、调用、执行、组合这些内容。Skill 的核心价值在于：

> **把组织经验变成 Claude 能真正调用、执行、验证和复用的系统资产**

关键洞察：很多团队用上 AI 后会发现——AI 会**放大组织已有的能力**，也会**放大能力缺口**。如果关键经验还没有 skill 化，AI 产出速度越快，组织的混乱程度也会越快。

---

## 能力地图总览

9 类 Skill 可以收束到**四个环节**，对应研发流程的不同阶段：

```
认知 ──→ 生产 ──→ 验证 ──→ 交付
```

| 环节 | Skill 类别 |
|------|-----------|
| 认知 | 内部库与接口文档、数据获取与分析、故障排查手册 |
| 生产 | 代码脚手架与模板、业务流程与团队自动化 |
| 验证 | 代码质量与审查、产品验证 |
| 交付 | 持续集成与部署、基础设施运维 |

---

## 第一环节：认知

**目标**：让 Claude 知道自己面对的是什么系统，从哪里获得理解，遇到异常时从哪里开始查。

### 1.1 内部库与接口文档（Internal Libraries & APIs）

**作用**：补充组织内部的专属知识

Claude 对通用技术知识本来就不差，但不了解：

- 内部库怎么用
- 内部平台怎么操作
- 内部命令行工具
- 内部设计系统

**Skill 最有价值的内容**：

- 最常用的调用方式参考片段
- 公开文档里没有写，但团队反复踩坑的坑点（Gotchas）
- 边界条件下的行为差异
- 常见误区总结

**核心原则**（原文反复强调）：

> 不要写 Claude 本来就知道的东西，要写它**推不出来**的东西。

**示例**：

- 某个内部计费库在边界条件下会出什么问题
- 某个前端设计体系最容易出现什么误区

**团队缺失信号**：这些知识全散在老员工脑子里、聊天记录里、零散文档里。

---

### 1.2 数据获取与分析（Data Access & Analysis）

**作用**：解决 Claude 到分析任务就空转的问题

很多团队已经让 Agent 写代码、写文档、写方案了，但一到分析任务就开始空转。问题往往在于**数据入口**。

**Skill 应整理的内容**：

- 数据源凭证
- 仪表盘 ID
- 常用查询语句
- 分析脚本
- 查询模板

**示例场景**：

- 漏斗分析
- 分群对比
- 监控面板接入

**团队缺失信号**：Claude 不知道去哪里拿数据，也不知道该用什么方式查。

---

### 1.3 故障排查手册（Troubleshooting Playbooks）

**作用**：接收症状 → 自动排查 → 输出结构化结果

这类 Skill 接收的输入是症状，比如：

- 一条 Slack 告警
- 一段错误签名
- 一个请求 ID

Claude 不只是给建议，而是会：

1. 自己去查日志
2. 查监控
3. 串链路
4. 输出结构化排查结果

**核心价值**：

承接**低频但认知负担极高**的问题。平时不一定天天用，但一旦出事，团队常常只能依赖少数最熟悉系统的人。

**本质**：把资深工程师的排查思路，变成组织可以复用的基础设施。

---

## 第二环节：生产

**目标**：让 Claude 稳定地产出代码、流程、结果和标准化产物，而不是每次都从零起步。

### 2.1 代码脚手架与模板（Scaffolding & Templates）

**作用**：给 Claude 一套稳定的骑手式

很多事情没必要每次从零搭，包括：

- 新建前端组件
- 新开接口
- 新建数据库迁移
- 新建工作流
- 新拉微服务模板

**没有这类 Skill 的问题**：

- Claude 每次都得重新决定目录结构
- 命名习惯不统一
- 文件分层混乱
- 默认股值飘忽不定
- 产出容易漂移

**有了这类 Skill 的效果**：

- 团队的结构和风格才有可能稳定下来
- 产出物一致性提高

---

### 2.2 业务流程与团队自动化（Workflow Automation）

**作用**：处理高频重复、规则清楚的流程工作

**示例场景**：

| 场景 | 自动执行内容 |
|------|-------------|
| 站会汇报 | 从任务系统、代码仓库、聊天工具汇总历史，自动整理成站会内容 |
| 事故报告 | 自动创建报告、同步状态、通知相关人 |

**核心价值**：

持续吃掉团队里那些**低价值重复性的劳动**。从组织效率角度看，这类 Skill 往往是最容易被快速感知到收益的一类。

---

## 第三环节：验证

**目标**：让 Claude 能**证明**自己写出来的东西真的能工作

这是原文认为**最值得重视**的一段。

### 3.1 代码质量与审查（Code Review）

**作用**：检查代码、文档或方案本身的质量

检查项包括：

- 安全风险
- 结构问题
- 命名习惯
- 性能隐患
- 团队规范
- 审查清单

**关键做法：Adversarial Review（对抗性审查）**

> 不要只让 Claude 用刚写完代码的那套视角回头审自己。

更好的做法：

1. **故意制造一个新鲜视角**，让它专门来挑毛病
2. 把问题修掉
3. 反复迭代
4. 直到剩下的只是一些小问题

**背后的原理**：对抗模型的**自我一致性偏差**（Self-consistency Bias）

Claude 刚写完代码，用的是同一套思路审查自己，容易遗漏问题。引入对抗性视角可以打破这种偏差。

---

### 3.2 产品验证（Product Verification）

**核心观点**：真正决定质量的不只是生成能力，还有**反馈闭环**。

这类 Skill 值得工程师花**一周时间专门打磨**。

**Skill 组成**：

- 无头浏览器
- TMUX（终端复用工具）
- 各种脚本和断言

**核心问题**：如何描述怎么测试代码是否真的正确工作？

**重要技巧：录制输出视频**

让 Claude 录制完整执行过程的视频，可以看到：

- 它到底测了什么
-点了什么
- 停在了哪里
- 有没有真的走完关键路径

**另一种做法：编程式断言**

把断言直接放进 Skill，让 Claude 在**每一步都强制检查状态**。

**示例**：

- 注册流程是不是完整跑通？
- 结账之后，发票是不是落在正确状态？

**核心目标**：让 Claude 有能力**证明自己做对了，而不只是交差**。

---

## 第四环节：交付

**目标**：让 Claude 把结果真正推进到系统里，并在后续阶段持续维护，而不是只停在"代码已经写完了"。

### 4.1 持续集成与部署（CI/CD）

**作用**：负责代码合并、流水线、部署、灰度、回滚这一整条交付链路

**Skill 应覆盖的场景**：

| 场景 | 自动化内容 |
|------|-----------|
| PR 合并 | 监控合并请求状态，重试不稳定的流水线，处理合并冲突，条件满足后自动合并 |
| 服务部署 | 灰度发布，盯关键指标，异常时自动回滚 |

**团队问题信号**：

> 一到部署，就必须完全切回人工操作

这说明 Agent 的能力链条只走到了**一半**，真正的交付链路还没有接上。

---

### 4.2 基础设施运维（Infrastructure Operations）

**作用**：处理基础设施和日常运维动作，尤其是**高风险动作**

**示例场景**：

- 查孤立资源
- 处理依赖升级
- 调查云资源费用异常
- 做容器集群故障处理
- 执行待确认点的清理流程

**特别强调：护栏（Guardrails）**

删资源、改环境、切流量——这些操作一旦失控，代价很高。

所以这类 Skill 更强调：

1. 先把**风险边界写清楚**
2. 再去谈自动化

**团队缺失信号**：基础设施操作依然高度依赖"谁熟谁来"，流程没有真正编码进系统。

---

## 能力地图的价值

> **9 类 Skill 地图更像是一张组织能力地图**

把团队现有的 Skill 摆上去，很快就能看出来：

- ✅ **哪些地方已经沉淀下来了**
- ❌ **哪些关键环节还停留在人脑里**
- ⚠️ **哪些地方明明最需要 Skill，结果到现在还没被系统化**

---

## 核心结论

原文真正讲的是：

> **团队落地 AI 研发时，真正的短板，往往在于关键经验有没有被做成 Claude 能真正调用、执行、验证和复用的 Skill。**

---

## 观众反馈补充

> **部署自己写自动脚本就好了**

这个反馈指出了交付环节的一个实践路径：对于持续集成与部署这类 Skill，很多团队确实可以通过自己编写自动化脚本快速落地，关键是要把脚本组织成可复用的 Skill 格式（包含配置、参数、边界条件等），而不是散落的独立脚本。
