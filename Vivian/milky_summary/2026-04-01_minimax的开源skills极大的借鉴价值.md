# MiniMax 开源 Skills 深度解析

## 核心观点

MiniMax 等模型厂商开始做 skills 开源，由于他们深度了解模型本身，这些 skills 具有极高的借鉴价值。

---

## Skills 编写两大核心原则

### 原则一：从底层出发处理文件

**Excel 为例的底层实现机制：**

| 层级 | 说明 |
|------|------|
| 用户层面 | Excel 文件（.xlsx） |
| 底层实际 | 压缩包格式（ZIP） |
| 内部结构 | 多个 XML 文件 |

**正确处理流程：**

```
Excel文件(.xlsx)
    ↓ 解压
XML文件集合（包含样式、内容、公式等）
    ↓ 编辑指定的XML文件
修改后的XML文件
    ↓ 重新压缩打包
最终Excel文件
```

**为什么这种方式更好：**

- 直接编辑 XML 避免了大模型对文件结构的理解偏差
- 可以精确控制文档的每个细节
- 编辑效果更稳定、更可控

### 原则二：Markdown 文档只写模型「不知道」的内容

**反面示例（❌ 常见错误）：**

- 详细描述「打开网站 → 输入关键词 → 点击搜索」这类常识流程
- 告诉模型 Excel 是什么、Word 是什么
- 写大模型已经训练好的基础知识

**正面示例（✅ 正确做法）：**

- 记录模型执行任务时**反复犯的错误**
- 记录**踩过的坑**和对应的解决方案
- 写**独特的工作流程**（模型不知道的特定领域知识）

> 核心逻辑：大模型脑子里已经有了 Excel 的概念，你不需要重复告诉它。你需要告诉它的是：**它容易搞错什么**以及**应该怎么纠正**。

## 核心启示

**写好 Skills 的关键：**

1. **实际测试** — 不要凭空想象，要让模型真正执行任务
2. **记录重复错误** — 如果同一个坑踩了 2-3 次，就一定要记录到文档中
3. **底层优先** — 处理 Office 文档时，优先考虑从 XML 层面操作
4. **聚焦未知** — 文档只写模型不知道的内容，而不是它已经掌握的知识

> **核心公式：优秀的 Skills = 大量测试 + 忠实记录踩坑 + 只写模型不知道的东西**
