# Claude Code 开发者踩坑经验分享

## 核心理念：让智能体像人一样思考

智能体工具设计的根本原则是**让智能体按照人的方式方法去操作**，而不是强加给它我们认为好的工具。工具应该根据模型能力不断演进：
- 基础工具：纸张、手动计算、计算器
- 高级工具：电脑、编程代码
- 工具复杂度与所需技能成正比

## 用户交互方式的演进

### 第一代方案：退出计划工具（失败）
- 实现方式：提供"退出计划"工具，使用纯文本交互
- 问题：让 Claude 感到困惑，违背常识逻辑
- 原因：要求 Claude 既提供计划又回答计划相关问题，逻辑矛盾

### 第二代方案：结构化输出（部分成功）
- 实现方式：固定计划后输出 Markdown 格式，开发者解析后让用户回答
- 问题：输出格式不稳定，经常不按格式输出导致解析失败

### 第三代方案：`ask_user_question` 工具（成功）
- 实现方式：专门的工具来询问用户问题和选项
- 优势：
  - 符合人类习惯和逻辑
  - Claude 可在任何时候调用
  - 强化了"完成任务→询问用户"的自然流程

**关键原则**：工具必须符合逻辑和人类习惯，而不是技术上的便利性。

## 任务管理系统的升级

### 从 To-Do List 到 Task 系统

#### To-Do List 的局限性
- 适用场景：早期模型指令遵循能力较弱时需要
- 问题：
  - 没有持久化存储
  - 无法在多智能体间传递
  - 现代模型（如 GPT-4）能力提升后变得过于苛刻
  - 缺乏灵活性，即使需要修改也会严格按原计划执行

#### Task 系统的优势
- **实现方式**：将任务计划写入共享的 Markdown 文档
- **核心特性**：
  - 多智能体可共享、修改、读取同一文档
  - 支持多智能体协作
  - 保持灵活性，允许计划调整
- **适用模型**：指令遵循能力强的现代模型

## 上下文获取策略

### RAG 方式的问题
- **稳定性差**：在各种环境下表现不稳定
- **逻辑违背**：预先获取的上下文不是模型自主查找的
- **用户体验差**：像是"脑子里突然多了奇怪的参考文献"
- **效果不佳**：不符合人类思考方式，而大模型基于人类语料训练

### Grep 关键词检索方式
- **实现原理**：模仿程序员的工作方式，使用关键词检索
- **操作流程**：
  1. 让智能体自己搜索文件
  2. 搜索关键函数库
  3. 自行构建连贯的上下文
- **优势**：
  - 上下文连贯性强
  - 智能体可控制获取过程
  - 支持渐进式披露理念
  - 不够时可继续获取

### 模型能力演进的影响
Claude 在一年多时间内的能力提升：
- **初期**：无法自己构建上下文
- **现在**：可以跨文件嵌套检索，精确找到所需上下文

## Skills 和渐进式披露

### Skills 工作机制
1. 先读取元数据
2. 判断该 Skills 是否为完成任务所需
3. 如果需要，则查看具体 Skills 内容
4. 执行相应功能

### 渐进式披露的实现方式

#### MCP（Model Context Protocol）方式
- **实现**：将工具作为标识符放置，需要时智能体自主查看
- **优势**：MCP 提示词不占用上下文，放在系统提示词中
- **搜索机制**：使用搜索方式定位 MCP 工具

#### 子智能体方式
- **适用场景**：只需要结果，不需要中间信息时
- **实现**：启动子智能体获取特定结果
- **原则**：只传递最终结果，不传递中间过程信息

### 工具数量控制
Claude Code 目前约有 20 个工具，每添加一个工具都需要精细评估：
- 是否真正需要
- 是否应该添加
- 对整体性能的影响

## 总结

智能体构建是一门**艺术而非科学**。核心要点：

1. **符合人类操作习惯**：工具设计必须遵循人类逻辑和思维方式
2. **避免想当然**：不要把智能体当作纯粹的工具，要考虑其思维模式
3. **持续演进**：根据模型能力提升不断调整工具和策略
4. **渐进式披露**：按需提供信息，避免上下文污染
5. **逻辑一致性**：确保工具调用流程符合自然逻辑

这些经验对所有编程智能体开发者都有重要参考价值，包括 OpenCursor、Menus 等项目都在沿用类似的设计理念。
