## 一、OpenClaw 简介与核心优势

### 1.1 什么是 OpenClaw

OpenClaw 是一个**开源项目**，采用 MIT License，可以免费使用，支持商业应用和代码修改。

### 1.2 三大核心优势

| 优势 | 说明 |
|------|------|
| **主动性** | 只需给工具和权限，AI 自动完成目标，无需手动定义工作流 |
| **持久记忆** | 记忆保存在本地，通过持续对话逐渐了解用户偏好，越用越精准 |
| **聊天软件集成** | 可通过 Telegram、WhatsApp、Discord 等直接远程指令和接收结果 |

## 二、机器选择与系统配置

### 2.1 云服务器 VS 本地机器

**推荐使用本地机器**，原因如下：

| 对比项 | 云服务器 (VPS) | 本地机器 |
|--------|---------------|----------|
| 优点 | 24小时不间断、完全隔离 | 完美依赖权限、可使用真实浏览器和本地APP |
| 缺点 | 无法使用高价值平台（YouTube、X、B站、抖音等），无头浏览器容易被封禁；无法操作本地文件和浏览器 | 需要设备保持开机 |

**核心逻辑**：OpenClaw 的能力完全取决于给的权限。云服务器无法使用本地浏览器和文件，功能大打折扣。

### 2.2 不同设备的选择方案

#### Windows 电脑

**推荐方式**：使用 WSL（Windows Subsystem for Linux）

```bash
# 一行命令即可安装 Linux 系统到 Windows 中
```

**两种 Windows 方案**：
1. 安装独立 Linux 系统，专门作为 OpenClaw 机器
2. 使用 WSL（适合不想重装系统的用户）

|----------|------|

#### 无旧电脑、只有一台电脑

- **Windows**：安装 WSL，务必做好文件隔离
- **Mac**：建议购买最便宜的 Mac mini 作为 OpenClaw 专用机器

### 5.2 方法一：内置 fetch 工具

**原理**：服务端 HTML 抓取 + 正文提取，用 HGP 请求获取内容，剥离广告、导航栏等噪音，输出干净 Markdown。

**优点**：
- 速度快、成本低
- 不需要启动浏览器
- 适合 24 小时长期扫描
- 更安全，不影响你的账户登录态

**缺点**：
- 无法执行 JavaScript
- 无法登录、交互、翻页、滚动、点击
- 容易被反爬和地域限制

**适用场景**：
- 监控竞争对手网站定价、新品发布、政策调整
- 追踪 AI 官方文档更新
- 公开文章（Wikipedia、News 等）

**配置**：无需配置，安装后自动可用

### 5.3 方法二：Social Fetch（API 方法）

**功能**：抓取 X (Twitter)、Reddit、YouTube 上指定时间范围内的爆点、痛点、趋势信息。

**信息来源**：
| 平台 | 数据来源 |
|------|----------|
| X | XAI 网络搜索功能 |
| Reddit | OpenAI 网络搜索功能 |
| YouTube | YT-DLP 开源库 |

**算法原理**：
1. 广撒网搜索关键词
2. 对高频关键词聚焦二次抓取
3. 按出现频率和互动信号加权总结

**优点**：
- 覆盖三大主流社交媒体
- 可交叉验证信息
- 轻量级，无需浏览器

**缺点**：
- 需要 API Key（付费）
- 只能做情报获取，无法执行浏览器操作

**每次搜索成本**：约 **0.2 美元**（包含 OpenAI + XAI 费用）

> 💡 **优化建议**：避免搜索 "AI" 这类广泛词汇，用 "openclaw setup" 这类精确关键词效果更好。

## 六、降低 Token 消耗的四种方法

### 6.1 消耗根源分析

每轮对话的输入包含：

1. **系统提示词** - AI 模型说明书
2. **Workspace 文件** - agent.json、tools、memory 等
3. **对话历史** - 滚动雪球效应，越来越长
4. **工具输出** - 抓取的网页内容、日志等
5. **当轮问题** - 你的具体问题

**问题本质**：每次对话都相当于把员工手册、公司章程、岗位 SOP 全部念一遍 → 成本极高。

### 6.2 方法一：使用 QMD（Query-Markdown-Description）

**原理**：建立本地 Markdown 知识库索引，查询时只返回最相关的片段，而非全文。

**工作流程**：

| 阶段 | 操作 | 成本 |
|------|------|------|
| 索引刷新 | 检测文件新增/修改/删除，更新分段路径和元数据 | 本地 |
| 向量更新 | 新增片段生成向量，投射到向量数据库 | 本地 |
| 语义检索 | 计算向量相似度，提取相关片段 | 云端 |
| 最终注入 | 只把命中的片段注入给 OpenClaw | 云端 |

**核心优势**：将云端 token 消耗转化为本地计算成本。

**安装步骤**：

```bash
# 方式一：让 OpenClaw 自动安装
# 方式二：手动安装
```

**配置**：在 `openclow.json` 中确保 memory 配置正确。

**精细控制参数**：

```json
{
  "limit": {
    "maximum_result": 5,      // 最多注入几段
    "maximum_snap": 1000,     // 每段允许的最大长度
    "maximum_inject": 5000    // 每轮总注入上限
  }
}
```

### 6.3 方法二：用本地模型跑心跳

**什么是心跳（Heartbeat）？**

OpenClaw 的定时唤醒机制：
- 按配置的频率触发 agent 执行心跳清单
- 先读取 `heartbeat.md` 文件
- 判断是否需要执行维护、提醒或任务
- 没事输出简短确认，有事执行任务

**心跳的作用**：
| 应用 | 说明 |
|------|------|
| 监工模式 | 定期刺激 agent，防止长期任务中途停止 |
| 定时任务 | 每天固定时间执行特定任务 |
| 状态检查 | 监控任务进度 |

**成本问题**：

| 心跳频率 | 每月触发次数 |
|----------|--------------|
| 每 30 分钟 | 1440 次 |

每次心跳都携带完整上下文（系统提示词、workspace 文件、对话历史），成本累积很高。

**解决方案**：用本地小模型跑心跳

```bash
# 下载 Ollama
# 根据内存配置选择模型（如 Qwen 各参数版本）
```

心跳只用来触发，不用来执行任务，低智商任务交给本地模型。

### 6.4 方法三：尽量使用订阅而非 API

| 方式 | 成本 | 稳定性 | 支持情况 |
|------|------|--------|----------|
| **OpenAI 订阅** | 较低 | 较高 | ✅ 支持 |
| **Claude 订阅** | 低 | 高 | ❌ 不支持，有封号风险 |
| **Google 订阅** | 低 | 高 | ❌ 不支持 |
| **API 按量付费** | 可能很高 | 最高 | ✅ 支持 |

### 6.5 方法四：生成成本审计报告

让 OpenClaw 自行分析消耗来源：

```
指令："生成一份成本审计报告，列出消耗最大的任务前5名，用百分比表示"
```

**常见浪费场景**：

| 问题类型 | 说明 |
|----------|------|
| 轻任务携带大上下文 | 简单任务消耗过多 token |
| 轮询代替事件触发 | 本应是条件触发，却做成每 N 分钟检查一次 |
| 重复执行相似任务 | 多个 agent 做重复工作 |

**优化方向**：

| 方面 | 具体措施 |
|------|----------|
| **流程优化** | 轮询任务改为事件触发 |
| **上下文优化** | 使用 QMD 减少注入量 |
| **模型优化** | 轻任务用便宜模型或本地模型 |
| **任务合并** | 合并重复任务 |

## 七、AI 团队搭建

### 7.1 Agent 协作原理

**Agent（智能体）** = 具备自主决策能力的 AI 程序，能够：
- 接收任务
- 规划执行步骤
- 调用工具完成任务
- 返回结果

**协作模式**：

- **串行协作**：Agent A 完成 → 触发 Agent B
- **并行协作**：多个 Agent 同时处理不同任务
- **层级协作**：主 Agent 分解任务 → 分发给子 Agent

### 7.2 案例：六人 AI 团队配置

#### Agent 1：情报员（每日 9 点触发）

| 配置项 | 内容 |
|--------|------|
| **触发时间** | 每天早上 9 点 |
| **任务** | 搜索 AI 技术进展、产品动态、用户痛点 |
| **对话规则** | 不回复 → 明天继续；回复 → 深挖该话题 |

#### Agent 2：私人助理（每日 7:30 触发）

| 配置项 | 内容 |
|--------|------|
| **触发时间** | 每天早上 7:30 |
| **功能** | 播报天气、今日待办（按重要性排序） |
| **日常使用** | 随时通过 Telegram 记录待办事项、查询进度 |
| **替代工具** | 完全替代 Notion 等笔记软件 |

#### Agent 3：内容总监（每隔 3 小时触发）

| 配置项 | 内容 |
|--------|------|
| **触发频率** | 每 3 小时 |
| **任务** | 搜索 X 上关于 AI/OpenClaw/AI模型 等话题的热门帖子 |
| **交互逻辑** | 👍 → 深挖背景信息；👍👍 → 提供视频封面和脚本建议；👁️ → 收藏到资料库 |
| **扩展功能** | 同时监控 YouTube 和 Hacker News |
| **用途** | 视频选题辅助系统 |

#### Agent 4：开发者（24×7 运行）

| 配置项 | 内容 |
|--------|------|
| **运行时间** | 几乎不间断 |
| **任务** | 开发任务 |
| **空闲时** | 自动测试、寻找边缘情况 bug |
| **适用场景** | 高价值工作（如开发、销售线索挖掘） |

> ⚠️ **24×7 运行成本极高**，必须确保任务价值足够高。

#### Agent 5：视频转文章助手（按需触发）

| 配置项 | 内容 |
|--------|------|
| **触发方式** | 手动丢视频链接 |
| **任务** | 将 YouTube 视频转换为带截图的文章 |
| **用途** | 内容再利用 |

#### Agent 6：机动成员（按需触发）

| 配置项 | 内容 |
|--------|------|
| **触发方式** | 给任务才执行 |
| **任务** | 临时性工作 |

### 7.3 多 Agent 协作最佳实践

| 原则 | 说明 |
|------|------|
| **明确分工** | 每个 Agent 有单一职责 |
| **合理触发** | 高频任务用定时触发，低频任务按需触发 |
| **成本控制** | 24×7 只用于高价值任务，轻任务用小模型 |
| **信息共享** | 通过共享文件夹或 memory 文件传递上下文 |
| **监控日志** | 使用可视化控制中心监控各 Agent 状态和成本 |

### 8.2 旧 Mac 升级建议

> **问题**：2012 年的 MacBook Pro A1278 能否运行 OpenClaw？

**解决方案**：使用 OCLP（OpenCore Legacy Patcher）可以安装 macOS 15+，让老旧 Mac 也能使用最新系统功能。

### 8.3 安全提醒

- 使用 OpenClaw Relay 插件时务必小心，操作失败可能直接影响你的网站登录态
- 给予 OpenClaw 的权限越大，能力越强，但风险也越高
- 建议从最小权限开始，逐步根据需要增加
