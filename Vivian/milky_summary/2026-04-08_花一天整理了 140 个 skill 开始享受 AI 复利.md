# 140+ Skill 管理实战：AI 复利工作法

## 核心理念：把 AI 使用变成可积累的资产

### 柳比歇夫工作法的启发

苏联昆虫学家柳比歇夫从 26 岁开始记录每天时间花费，精确到分钟，持续记录 56 年直到 82 岁去世。这套方法的核心原则是：

> **只记录不评判，凡事先记下来再说**

将这一理念迁移到 AI 使用场景：你的每一个 AI 对话、每一次 skill 创建，都是你的 **AI 时间日志**。通过分析这些日志，可以：
- 了解自己在哪些环节高频使用 AI
- 评估 AI 使用效果
- 持续优化 AI 工作流

### 为什么 skill 管理如此重要

当你的 skill 数量达到 100 个、1000 个甚至 1 万个时：

```
个人能力边界 ← 突破 → 时间和精力的限制
```

多个 agent 同时工作可以突破个人能力上限，而 skill 就是管理这些 agent 的标准化方式。

### 写作类 Skill（30+ 个）

#### 1. 社交媒体动态生成
- **用途**：将长文转化为社交媒体动态
- **操作**：输入一篇长文 → AI 模仿个人风格生成多条动态 → 删减发布
- **适用场景**：极客时间、微信公众号、X（Twitter）等平台

#### 2. 播客内容处理
- **用途**：将几万字的访谈内容自动总结
- **工作流**：
  ```
  播客音频 → 字幕提取 → 内容提炼 → 核心观点总结 → 飞书卡片
  ```

#### 3. 前端管理界面
由于写作类 skill 数量众多，作者制作了专门的管理界面，实现：
- 可视化选择不同写作场景
- 一目了然知道该用哪个 skill

### 信息类 Skill

#### 1. RSS/博客自动总结
- **工具链**：RSS 源 → AI 总结 → 飞书推送
- **体验**：作者形容为「皇帝批奏折」般的快感

```
视频/博客 → 自动字幕提取 → 转写文章 → 传送到飞书 → 转成卡片 → 发布到网站
```

## Skill 创作方法论（4 种方法）

### 方法一：反推法（从交付物入手）

**适用场景**：有明确的输出标准

**操作步骤**：

1. 找一份你最满意的交付物（PRD、周报、分析报告、视频脚本等）
2. 将这份交付物发送给 AI
3. 让 AI 按照这个文件标准讨论输出类似内容
4. 两人（你+AI）协作完成一份新内容
5. 将整个过程制作成 skill
6. 下次直接调用，一致输出

```
满意交付物 → 给 AI 学习 → 协作创作 → 封装为 skill → 批量产出
```

### 方法二：流程法（从工作流入手）

**适用场景**：复杂任务需要多个步骤

**操作步骤**：

1. 将复杂任务拆解为多个独立 skill
2. 分别测试每个 skill，确保稳定运行
3. 逐步串联，形成完整工作流
4. 最终合并为一个综合 skill

**示例**：博客处理流程
```
拉取更新 → 选择播客 → 字幕提取 → 内容提炼 → 存飞书 → 发布
```
每个环节单独 skill → 最终合并成 `博客自动处理` skill

### 方法三：归纳法（从聊天记录入手）

**核心思路**：有时候不知道想要什么，但可以从历史对话中发现

**操作步骤**：

1. 回顾最近与 AI 的聊天记录
2. 识别高频需求（如：每次都让 AI 取标题）
3. 如果发现某个提示词重复出现 3 次以上
4. 立即将其封装为 skill

```
聊天记录 → 发现重复需求 → 封装为 skill → 告别重复输入
```

### 方法四：经验提炼法（从隐性知识入手）

**适用场景**：无法言传只可意会的经验

**内容类型**：
- 个人写作风格
- 审美标准
- 公司代码规范
- 团队设计原则

**操作步骤**：

1. 初始阶段就建立基础的 skill 框架
2. 在使用中逐步积累和优化
3. 每次不满意时让 AI 修复并记录
4. 渐进式完善 skill

## Skill 使用效率提升（7 个办法）

### 办法一：核心知识写入 MD 文件

**推荐**：将常用信息写入 `dot.md` 或 `cloud.md`

```markdown
# cloud.md 示例
## 我的写作风格
- 语言简洁，避免冗余
- 使用短句，每段不超过 3 行
- 善用数据和案例支撑观点

## 审稿标准
- 检查事实准确性
- 确保逻辑连贯
- 标题要吸引人
```

**使用方式**：每次调用 AI 时提示：
> 「请根据 cloud.md 中的写作风格和审稿标准处理」

**效果**：`cloud.md` 可以被 AI 严格遵守执行

### 办法二：制作管理界面

**适用场景**：skill 数量超过 10 个

**问题**：数量多导致记不住、选择困难

**解决方案**：开发前端界面
- 可视化展示所有 skill
- 根据场景分类呈现
- 一键调用

### 办法三：自动化轻量化任务

**原则**：不需要人工参与的重复任务，设置为全自动

**示例**：
```
播客处理 → 完全自动化 → 推送飞书 → 发送网站
人工介入：0
```

### 办法四：移动端处理轻任务

| 平台 | 适合的任务 |
|------|------------|
| **手机** | RSS 更新、消息推送、轻量笔记 |
| **电脑** | 代码修改、原型制作、需要浏览器预览的工作 |

**应用示例**：
- RSS 订阅 → 推送到飞书 IM → 手机查看
- 重度工作 → 电脑端处理

### 办法五：GitHub 统一管理

**优势**：
1. 换电脑也能使用同一套 skill
2. 版本控制，随时回滚

**操作**：
```bash
# 将 skill 仓库同步到新设备
git clone https://github.com/your/skill-repo.git
```

### 办法六：软链接同步（跨平台访问）

**问题**：使用多个应用（Cursor、VS Code、Open Cloud）需要访问同一个 skill 目录

**解决方案**：使用软链接（相当于 Windows 快捷方式）

```bash
# Linux/Mac
ln -s /path/to/master/skill /path/to/app/skill

# Windows
mklink /D C:\link\to\skill D:\master\skill
```

### 办法七：API 安全管理（环境变量）

**问题**：skill 中直接写入 API Key 会导致泄露风险

**解决方案**：使用环境变量

```bash
# 设置环境变量（Linux/Mac）
export OPENAI_API_KEY="your-api-key"

# Windows PowerShell
$env:OPENAI_API_KEY="your-api-key"

# 在代码中使用
import os
api_key = os.environ.get('OPENAI_API_KEY')
```

**绝对禁止**：将 API Key 直接写在代码文件或 prompt 中

### 办法八：Skill 自迭代机制

**原理**：让 skill 在使用中自我进化

**操作流程**：

1. 在 skill 中添加「自我修复」能力
2. 用户对交付物不满意时，让 AI 修复
3. AI 将修复内容记录并更新到 skill 本身
4. 下次使用自动应用新版本

```
用户反馈 → AI 修复 → 记录修复内容 → 更新 skill → 下次自动应用
```

**效果**：每个 skill 使用一次就进化一次，长期复利

## 观众补充与最佳实践

### 常见陷阱

| 陷阱 | 表现 | 解决方案 |
|------|------|----------|
| skill 太多不用 | 创了 100 个只用了 3 个 | 定期复盘，删除低频 skill |
| 忘了用过某个 skill | 重复造轮子 | 做好分类和命名 |
| skill 质量参差不齐 | 输出不稳定 | 建立自己的 skill 模板 |
